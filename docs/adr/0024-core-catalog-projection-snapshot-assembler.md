# Core Catalog separates admin read detail from sales-channel projection snapshot

Core Catalog will not use the 관리자 version detail read model as the direct source for `ProductMasterActiveVersionChanged` snapshots. Version-owned read fragments may be shared, but 관리자 version detail and the 판매채널 Projection Snapshot are assembled by separate modules because draft/inactive authoring views tolerate incomplete data while sales-channel projection is an active-version contract consumed by channel-adapter without Core API fallback.

The 판매채널 Projection Snapshot is generated only for an active 판매상품 version. It must fail instead of emitting a snapshot when required projection data is incomplete, including missing calculated price projection or zero variants. Product image references are treated canonically as File UUIDs, while the existing snapshot contract remains compatible with current consumers, including legacy URL-shaped fields where required.
`ProductSnapshot.thumbnail` should carry the primary image File UUID, not a file-service URL. Current channel-adapter/storefront thumbnail rendering can normalize File UUIDs to public file URLs, but the implementation slice should verify that compatibility and separately track any direct URL consumers such as OpenGraph metadata or detail-image fallback paths.
`ProductSnapshot.images[].url` remains present for event-contract compatibility, but in the first slice it should carry the same File UUID value as `images[].fileId`. Core Catalog should not synthesize file-service URLs for product image projection fields.

Publish workflow should perform validations and price projection materialization before changing active state, then assemble and enqueue the sales-channel projection snapshot only after the target version is active.
Snapshot assembly and outbox enqueue stay inside the same publish transaction as the active status change. If sales-channel Projection Snapshot assembly fails after the target version has been set active in the transaction, the publish transaction rolls back rather than leaving an active version without the corresponding channel-adapter event.
Publish-time price projection materialization must refresh existing cache rows for the target version instead of preserving stale rows with insert-only conflict handling. Re-publishing an inactive version or publishing after pricing changes must snapshot prices calculated from the current version rules.
`ProductMasterActiveVersionChanged.changeReason` should be derived from the target version's status before publish: `draft` means `published`, and `inactive` means `rollback`. Whether there was a previous active version is a separate payload fact carried by `previousActiveVersionId`, not the basis for distinguishing publish from rollback.
The event-building method should receive the caller-determined `changeReason` explicitly instead of inferring it from `targetStatus` or `previousActiveVersion`. Publish/unpublish callers hold the transition context needed to choose the correct reason.
Active-version policy resync paths, such as membership price visibility and members-only visibility updates, continue to emit the existing `published` change reason in the first slice so the event contract does not expand. A more precise reason such as active projection update/resync is a separate event-contract migration.
That reason-enum cleanup is only recorded as follow-up context for now; it does not need a separate issue until the consumer-facing contract migration is designed.

The first implementation slice is limited to the Projection Snapshot assembler, shared version-owned read fragments, and the publish workflow connection. Legacy approval/bulk paths that bypass the publish workflow are known follow-up debt and should be fixed together in a later operations workflow pass rather than mixed into the first slice.

Shared version-owned read fragments should follow the existing product `loaders/` pattern. Loaders expose DB-visible version-owned facts, while 관리자 detail assembly and 판매채널 Projection Snapshot assembly remain separate assembler responsibilities.

Read fragment loaders should not enforce Projection Snapshot validity. They return reusable DB facts for a master/version, and the 판매채널 Projection Snapshot assembler owns active-version validation such as active status, non-empty variants, and required calculated prices.

The first implementation slice should wire the shared loaders into both the 판매채널 Projection Snapshot assembler and the existing 관리자 detail assembler where their read fragments overlap, while preserving the 관리자 detail response shape.
Add a single `ProductVersionReadLoader`-style facade for version-owned product fragments that are currently private helper queries, such as version row, categories, variants, images, and purchase constraints. Keep existing focused loaders such as `OptionReadLoader` and `TagReadLoader` rather than folding them into the new facade.
The new loader facade returns neutral DB-fact fragments, not 관리자 DTOs or event-contract shapes. Derived fields such as thumbnails, legacy image URLs, status remapping, option display formatting, and Projection Snapshot requiredness stay in the assemblers that own those contracts.
Calculated price projection stays outside the product version fragment loader. Assemblers use the pricing module's `VariantPriceCacheService` directly because 가격 is not a core 판매상품 fact: 관리자 detail may attach cached prices optionally, while the 판매채널 Projection Snapshot assembler treats missing required calculated prices as a publish-blocking validation failure.
Locale-sensitive display fragments should support an explicit locale with `ko-KR` as the default. The first implementation slice does not add a caller path for passing locale into sales-channel Projection Snapshot assembly, so active snapshot events continue to resolve display strings through the default `ko-KR` fallback.
This locale support is internal to loaders/assemblers in the first slice. The `ProductSnapshot` event contract keeps its existing single display-string shape rather than expanding to locale-keyed maps; channel-adapter continues to consume `ko-KR` fallback strings until a separate contract migration is designed.
Category fragments in a 판매채널 Projection Snapshot must be scoped by both master id and version id. The snapshot includes the mapped categories with their `isActive` and `visibility` flags rather than filtering them out in Core; sales-channel projection logic decides how those flags affect customer-facing category visibility.
The category fragment should include mapping facts such as `isPrimary` even though `ProductSnapshot.categories[]` does not expose that field. The event payload builder derives `categoryIds`, `primaryCategoryId`, and snapshot category objects from the same version-scoped fragment, avoiding a separate primary-category query and preventing master-only category mappings from leaking into active snapshots.
If a version has category mappings but no primary category, the event may carry `primaryCategoryId: null`. More than one primary category mapping for the same master/version is invalid data and should fail snapshot/event assembly rather than choosing one arbitrarily.
The Projection Snapshot assembler may return a small assembly result containing the `ProductSnapshot` plus category projection facts such as `categoryIds` and `primaryCategoryId`. It should not construct the full `ProductMasterActiveVersionChanged` payload; event metadata such as `changeReason`, `previousActiveVersionId`, and `changedAt` remains the event builder's responsibility.
Name this narrow result shape `ProjectionSnapshotAssembly` and limit it to `snapshot`, `categoryIds`, and `primaryCategoryId`. Do not add event metadata such as `changeReason`, `previousActiveVersionId`, `changedAt`, aggregate identifiers, or outbox topic fields to this type.
The Projection Snapshot includes only active variants from the active version. Draft, inactive, archived, or deleted variant states are not 판매채널 projection inputs; changing sellable variant status is a versioned product change that should reach sales channels through a later publish/rollback. If an active version has no active variants, snapshot assembly fails.
Each active variant in a Projection Snapshot must have a calculated price cache row for the active version. `basePrice` and `membershipPrice` are required numeric projections; `tieredPrices` may be absent semantically and should be normalized to an empty array. Missing cache rows are operator-fixable publish-blocking data problems, while non-numeric required price fields indicate a broken invariant.
Because the Projection Snapshot assembler is active-snapshot-only, `ProductSnapshot.status` is always `active`. If the target version is not DB-visible as active when the assembler runs, assembly fails instead of remapping `inactive` or `draft` into another snapshot status.
Projection Snapshot option groups and option values should be reduced to the option schema actually used by the active variants included in the snapshot. 관리자 detail can show the full version option schema, but sales-channel projection should not expose option values that have no sellable active variant behind them.
For active variants that use option values, the Projection Snapshot assembler requires non-empty locale-resolved display strings for the option group and option value. Missing or blank display rows for the default `ko-KR` locale are publish-blocking data problems; the sales-channel snapshot must not fall back to UUIDs or empty strings for customer-facing option labels.
Optionless products remain optionless in the Core Projection Snapshot: `optionGroups` is empty and active variants have an empty `optionCombination`. Adapter-specific default option/default option value synthesis, such as Medusa's requirement for optioned variants, stays in channel-adapter rather than becoming Core Catalog state.
Projection Snapshot tags include only version-mapped tag values whose tag value and tag group are active. Inactive tag metadata is not emitted as customer-facing sales-channel projection data.
If a version has no purchase constraint mapping, Core keeps `ProductSnapshot.purchaseConstraint` absent rather than changing the event contract to require `null`. Channel-adapter remains responsible for projecting absence as `null` into Medusa metadata so stale purchase-constraint metadata is cleared.

Projection Snapshot validation failures that represent operator-fixable product data should follow Core's existing HTTP exception handling rules and surface as explicit 400-level errors, using the Catalog convention of `BadRequestException` for publish-blocking data problems. Generic `Error` is reserved for broken invariants or unexpected infrastructure/code failures that should be treated as 500-level failures by the global exception filter.

All active `ProductMasterActiveVersionChanged` events should use the Projection Snapshot assembler, not only `publishVersion()`. Active-version policy updates that intentionally resync the sales-channel projection, such as membership price visibility or members-only visibility changes, must go through the same snapshot contract and validation.
The Projection Snapshot assembler is responsible only for active snapshots. `unpublished` events remain an event-payload concern: they carry no snapshot, no active version id, and no category projection because they represent "no active version" rather than an alternate snapshot shape.

Focused verification for the first slice should cover three Core layers: the shared product version read loader, the Projection Snapshot assembler, and the publish/event orchestration in `ProductVersionsService`. Loader tests should protect version scoping and neutral fragment behavior. Assembler tests should protect active-only validation, active variant filtering, price projection requiredness, option display requiredness, and UUID image projection. Publish/event tests should protect ordering, transaction-scoped outbox enqueue, `draft` to `published` versus `inactive` to `rollback` reasoning, and active policy resync through the same assembler. Channel-adapter transformer tests should also cover `thumbnail` and `images[].url` carrying File UUIDs.
