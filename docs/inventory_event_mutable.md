# 재고 관리에 있어 event와 mutable state의 적절한 사용 (동영)

## Append only 테이블

지난 번부터 소개해 드린 설계에서, 재고 관리를 이벤트 테이블과 재고
테이블로 분리해서 설명드렸습니다. 해당 설계의 핵심은 이벤트 테이블에는
언제나 append만이 일어난다는 점입니다. 예를 들어 입고가 발생하여 재고를
증가시켰는데, 이것이 착오에 의한 것임을 알게 되어 입고를 취소해야 하는
상황에, 입고 이벤트를 삭제하는 대신 입고를 취소하는 이벤트를 하나 더
만들어 맨 뒤에 붙이게 됩니다. 시간이 흐름에 따라 발생하는 모든 사건을
이벤트로 기록하고 (핵심적인 데이터는) 수정하지 않는 이러한 방식을 Event
sourcing, 내지는 Append only 등 여러 이름으로 부릅니다.

\[이미지 1\]

  event   설명   stock    b    d
  ------- ------ -------- ---- ----
  e1      입고   p1 × 3   e1   e2
                 p2 × 2   e1   e2
  e2      출고   p1 × 1   e2   

이 예시는 `입고 → 출고` 과정을 통해 최종적으로 p1이 2개, p2가 2개 남는
상황을 보여줍니다. stock row에 생성(b)과 소멸(d) 시점을 기록함으로써
특정 시점의 재고를 쉽게 추적할 수 있다는 점이 핵심입니다.

------------------------------------------------------------------------

## Mutable state

mutable state는 회원 데이터나 블로그 포스트 테이블 따위에 사용되는
일반적인 CRUD라고 보실 수 있습니다. 필요하다면 데이터가 update되거나
delete될 수 있으며 현재 상태만을 기록하는, 저희가 가장 흔하게 보는
데이터입니다. 현재 상태만을 기록한다는 것은 과거의 데이터는 유실된다는
말에 다름없습니다. 그러니 재고를 이렇게 관리한다면 분명히 문제가
생깁니다.

그런데, 재고가 가질 수 있는 상태 중 `출고가능 - 출고대기 - 출고작업중`
상태를 보시면, `출고대기` 가 `출고작업중` 이 되는 조건이 피킹이라는 걸
보실 수 있습니다. 큰 피킹 카트를 몰고 랙 사이를 돌아다니면서 상품에
바코드를 한 번 찍을 때마다 상품 하나의 재고 하나가 출고대기에서
출고작업중 상태가 되어야 합니다. Event sourcing 방식을 따르자면 바코드를
한 번 찍을 때마다 event 테이블에 row가 하나씩 생기고, stock 테이블에는
row 하나가 soft delete, 또 다른 하나는 insert됩니다. 그런 일이 출고작업
하나에 수백 번씩 이뤄진다고 생각하면 단순한 출고작업만으로도 추후 쿼리에
부담을 줄 정도로 row가 많아지고, 보기에도 아름답지 못합니다. 이런
상황에서는 단순히 재고를 나타내는 row를 직접 update해서 숫자를 1씩
늘리는 게 오히려 적절할 수 있습니다.

------------------------------------------------------------------------

\[이미지 2\]\
출고작업이 시작된 시점입니다. 기존 event sourcing 기반 재고
테이블(events, stocks)에서 **임시로 mutable state 기반
테이블(outbound_task_stocks)** 로 재고 일부(p1 30개, p2 50개)가
이동했습니다.

-   **events / stocks 테이블**: 여전히 p1 100개, p2 100개가 출고
    가능으로 표시됨\
-   **outbound_task_stocks 테이블**: t1 작업에 p1 30개, p2 50개가
    출고대기 상태로 기록됨

즉, 왼쪽만 보면 재고가 증발한 것 같지만 오른쪽까지 합치면 총량은
보존됩니다.

------------------------------------------------------------------------

\[이미지 3\]\
출고 작업자가 피킹을 시작했습니다. p1 상품 하나를 바코드로 찍으면 상태가
변화합니다.

-   **outbound_task_stocks**
    -   p1: 출고대기 29, 출고작업중 1\
    -   p2: 출고대기 50, 출고작업중 0

이 변화는 **이벤트 추가가 아니라 row update** 로 처리됩니다.

------------------------------------------------------------------------

\[이미지 4\]\
시간이 흘러 작업이 모두 끝났습니다. 모든 상품이 포장 완료되어 출고 작업
종료 버튼을 누르는 시점입니다.

-   **outbound_task_stocks**
    -   p1: 출고대기 0, 출고작업중 0, 출고완료 30\
    -   p2: 출고대기 0, 출고작업중 0, 출고완료 50

즉, 임시 mutable state 테이블에 있던 데이터가 commit 준비된 상태가
됩니다.

------------------------------------------------------------------------

\[이미지 5\]\
마지막으로 출고 완료 이벤트(e3)가 생성됩니다.

  event   설명            stock                상태
  ------- --------------- -------------------- ----------
  e1      초기조건        p1 × 100, p2 × 100   출고가능
  e2      출고작업 시작   p1 × 70, p2 × 50     출고가능
  e3      출고작업 종료   p1 × 30, p2 × 50     출고완료

-   **outbound_task_stocks 테이블**은 expired 상태가 되어 더 이상 재고에
    영향을 주지 않습니다.\
-   최종적으로 event sourcing 기반의 테이블에 `출고완료` 이벤트가 append
    되어 재고의 일관성이 유지됩니다.

------------------------------------------------------------------------

👉 이 과정을 통해 event sourcing의 안정성과 mutable state의 유연성을
동시에 가져갈 수 있습니다.\
- **event sourcing**: 이력 관리, 신뢰성 보장\
- **mutable state**: 실시간 작업 효율성 확보
