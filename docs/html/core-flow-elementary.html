Core 흐름 한눈에 보기

Core는 주문쪽지를 받아서, 창고가 실제로 움직일 수 있는 출고표로 바꾸는 큰 사무실이다.

한 문장으로 말하면

손님이 상품을 사면 Core는 “이 주문을 어떤 창고 물건으로 보내야 하지?”를 확인하고, 가능하면 출고표를 만들고, 모르면 대기줄에 세운다.

주문쪽지 상품 창고물건 출고표 재고예약

등장인물

판매상품: 손님이 쇼핑몰에서 고르는 상품.

창고물건: 창고에 실제로 있는 물건. 코드에서는 SKU.

연결표: 판매상품 1개가 창고물건 몇 개로 나가는지 적은 표.

출고표: 창고 직원이 보고 집어야 하는 일감.

SO, FO, FOI 먼저 알기

SO 판매주문

손님이 산 상품 목록이다.

젤네일 스타터 세트 x 1

DB: sales_orders, sales_order_lines

→
매칭표 바꾸는 규칙

판매상품 1개가 창고 SKU 몇 개로 나가는지 적은 표다.

베이스젤 SKU x 1
탑젤 SKU x 1
우드스틱 SKU x 2
파우치 SKU x 1

DB: product_matchings, product_variant_sku_links

→
FO / FOI 출고표

창고 직원이 실제로 집어야 할 SKU 목록이다.

베이스젤 SKU x 1
탑젤 SKU x 1
우드스틱 SKU x 2
파우치 SKU x 1

DB: fulfillment_orders, fulfillment_order_items

주문 수량이 곱해지는 방식

손님 주문 SO
젤네일 스타터 세트 x 3
×
출고표 결과 FOI
베이스젤 SKU x 3
탑젤 SKU x 3
우드스틱 SKU x 6
파우치 SKU x 3

그래서 fulfillment_order_items는 “손님이 산 판매상품”이 아니라 “창고가 꺼낼 SKU 줄”을 저장한다.

가장 큰 그림

1. 손님이 주문한다

쇼핑몰, 네이버, 쿠팡 같은 곳에서 주문이 들어온다.

OrderCreated 이벤트
→

2. Core가 주문쪽지를 저장한다

누가, 무엇을, 몇 개 샀는지 판매주문으로 적어 둔다.

sales_orders / sales_order_lines
→

3. 출고 대기줄에 넣는다

창고 출고표를 만들 일을 줄에 세운다. 실패해도 주문은 사라지지 않는다.

fulfillment_order_creation_backlogs

4. 연결표를 본다

판매상품이 어떤 창고물건으로 나가야 하는지 찾는다.

product_matchings / product_variant_sku_links
→

5. 출고표를 만든다

창고 직원이 집을 SKU와 수량으로 바꿔 적는다.

fulfillment_orders / fulfillment_order_items
→

6. 재고를 예약하고 보낸다

물건을 잡아두고, 피킹하고, 송장을 뽑고, 배송한다.

reservations / picking / invoice

시간 순서로 보기

주문 1건이 지나가는 길
1

주문 도착

외부 채널이 Core에게 “주문이 생겼어”라고 알려준다.

consumer: OrderEventsConsumer

2

주문 저장

Core가 주문과 주문상품 줄을 DB에 저장한다.

service: SalesOrdersService

3

출고 준비 줄

결제 완료 주문이면 출고표 만들기 줄에 들어간다.

status: pending

4

창고물건 찾기

판매상품을 창고물건 SKU로 바꿀 수 있는지 확인한다.

matching lookup

5

모르면 멈춤

연결표가 없으면 “나중에 다시 해” 하고 대기한다.

status: awaiting_matching

6

알면 출고표

SKU와 수량으로 창고용 출고표를 만든다.

FO / FOI 생성

7

예약과 배송

재고를 예약하고 피킹, 검수, 송장, 출고로 간다.

ready 또는 unfulfillable

Core가 갈림길에서 하는 판단

연결표가 있다

판매상품을 SKU로 바꿀 수 있다.

예: 상품 1개 = 창고물건 A 1개 + 창고물건 B 2개

결과: 출고표 생성

연결표가 없다

창고가 무엇을 꺼내야 하는지 모른다.

그래서 주문은 지우지 않고, 출고만 기다리게 한다.

결과: 매칭 대기

창고물건이 필요 없다

디지털 상품처럼 물리 배송이 필요 없는 상품일 수 있다.

그 줄은 출고표에 넣지 않는다.

결과: 물리 출고 제외

가게 말과 코드 말 연결하기

쉬운 말 코드/DB에서 부르는 말 하는 일
손님 주문쪽지 sales_orders 주문번호, 고객, 배송지, 총금액을 저장한다.
주문 안의 상품 줄 sales_order_lines 어떤 판매상품을 몇 개 샀는지 저장한다.
판매상품 variant 쇼핑몰에서 파는 실제 선택 단위다.
창고물건 SKU 창고에 실제로 보관하고 꺼내는 물건이다.
판매상품과 창고물건 연결표 product_matchings, product_variant_sku_links 판매상품 1개가 어떤 SKU 몇 개로 나가는지 알려준다.
출고 만들기 대기줄 fulfillment_order_creation_backlogs 출고표를 만들 수 있을 때까지 다시 시도하는 줄이다.
창고 출고표 fulfillment_orders 창고가 처리할 출고 작업 묶음이다.
출고표 안의 물건 줄 fulfillment_order_items SKU별로 몇 개 집어야 하는지 적는다.
물건 찜하기 stock_reservations 다른 주문이 못 가져가게 재고를 잡아둔다.

두 가지 예시

예시 A: 평범한 립스틱 주문

손님이 립스틱 2개를 산다.

Core가 연결표를 본다.

립스틱 판매상품 1개 = 립스틱 SKU 1개

그래서 창고 출고표에는 립스틱 SKU 2개라고 적힌다.

예시 B: 선물세트 주문

손님이 선물세트 3개를 산다.

Core가 연결표를 본다.

세트 1개 = 립스틱 SKU 1개 + 파우치 SKU 1개

그래서 창고 출고표에는 립스틱 SKU 3개, 파우치 SKU 3개라고 적힌다.

막히는 경우

문제

주문은 들어왔는데 판매상품이 어떤 창고물건인지 모른다.

→

Core의 행동

주문은 그대로 저장하고, 출고표 만들기만 대기시킨다.

awaiting_matching
→

풀리는 순간

관리자가 연결표를 저장하면 Core가 대기줄을 다시 깨워 출고표 생성을 재시도한다.

기억할 한 줄

Core는 주문을 바로 창고에 던지지 않는다. 먼저 “판매상품을 창고 SKU로 바꿀 수 있는지” 확인하고, 알면 출고표를 만들고, 모르면 주문은 살려 둔 채 출고만 기다리게 한다.
