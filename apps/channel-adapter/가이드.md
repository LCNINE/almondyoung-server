📌 Context
- 프로젝트: AlmondYoung Adapter 서비스
- 프레임워크: Nest.js + Drizzle ORM
- 목적: 외부 판매채널(네이버/쿠팡/자사몰)과 내부 OMS/WMS를 연결하는 어댑터 서비스
- 현재 구조:
  * channel-adapter.controller.ts : HTTP 엔드포인트 집합(폴링, 웹훅, sync-to, command, query)
  * channel-adapter.service.ts : 컨트롤러 → 오케스트레이터 파사드
  * adapter-orchestration.service.ts : 핵심 오케스트레이션 (pollAndPublish, handleIncoming, execute, syncToChannel 등)
  * sync-status.service.ts : 현재는 메모리 기반 동기화 상태 관리
  * null-event-publisher.service.ts : 개발/테스트용 이벤트 퍼블리셔
  * schema.ts : event_logs(외부 이벤트 기록), sync_histories(동기화 이력)
  * types.ts : DataType, InternalOrderEvent, ChannelCommand 등 표준 타입/DTO

🎯 Goal
- CTO의 SoT 원칙(“외부 SoT는 동기, 내부 SoT는 이벤트”)을 반영한 **리팩터링 가이드 문서(adapter-refactoring-plan.md)**를 작성해줘.
- 문서에는 아래 내용이 반드시 포함돼야 해:
  1. **현행 구조 요약**: 주요 Controller/Service/Orchestration의 역할, 호출 순서, event_logs/sync_histories 스키마 요약.
  2. **문제점/개선 포인트**: Consumer 부재, 멱등키 체계 미흡, SyncStatus 영속화 필요 등.
  3. **리팩터링 설계안**:
     - Consumer 추가 구조(파일 위치·토픽명·멱등키 체크 방식)
     - AdapterOrchestrationService 확장 포인트
     - SyncStatusService 확장 설계
     - ChannelStrategy 인터페이스 강화(updateStock/updateFulfillment 등)
     - 멱등키+버전 전략 및 DLQ/재시도 정책
  4. **파일별 수정/추가 목록**: controller/service/consumer/schema/types 별로 구체적 변경사항.
  5. **체크리스트**: 리팩터링 후 검증해야 할 항목(Jest 테스트, ESLint, DTO 강검증, 운영 모니터링 지표 등)

```ts
// 예: StockEventConsumer
@Injectable()
export class StockEventConsumer {
  constructor(private readonly orchestrator: AdapterOrchestrationService) {}

  @KafkaSubscribe('wms.stock.changed')
  async handle(evt: StockChangedEvent) {
    const key = `WMS:STOCK_CHANGED:${evt.sku}:${evt.eventVersion}`;
    if (await isProcessed(key)) return;

    await this.orchestrator.syncToChannel('all', {
      dataType: 'inventory',
      payload: { sku: evt.sku, deltaQty: evt.deltaQty },
    });

    await markProcessed({
      idempotencyKey: key,
      source: 'WMS',
      eventType: 'STOCK_CHANGED',
      resourceId: evt.sku,
      eventVersion: evt.eventVersion.toString(),
    });
  }
}
```
