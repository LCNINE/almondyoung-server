초등학생도 이해하는 결제 흐름

결제는 “돈을 바로 빼는 일”이 아니라, 약속표를 만들고 확인한 뒤 확정하는 일입니다.

이 페이지는 wallet 서비스의 결제 흐름을 쉬운 말로 풀어쓴 그림입니다. 어려운 단어는 먼저 생활 속 비유로 설명하고, 그 아래에 실제 코드 이름을 붙였습니다.

한 줄 요약 Intent는 “결제 예약표”, Charge는 “실제 돈을 움직여 본 기록”, Provider는 “결제 담당자”입니다.
단어 뜻 전체 흐름 결제수단별 차이 상태 뜻

먼저 단어부터 쉽게 보기

코드에 나오는 어려운 이름을 “학교 매점에서 물건 사는 상황”처럼 생각하면 됩니다.

관련 코드: apps/wallet/src/payment-intents
Payment Intent

결제 예약표

intent

“누가, 얼마를, 어떤 주문 때문에 결제하려고 하는지” 적어 둔 표입니다. 아직 돈이 빠져나간 것은 아닙니다.

예: “민수가 12,000원짜리 주문을 결제하려고 한다”라고 적힌 예약표
Charge

결제 시도 기록

charge

실제로 포인트를 잡아두거나, 카드사/은행에 결제를 부탁한 기록입니다. 포인트+카드 복합결제면 charge가 2개 생길 수 있습니다.

예: “포인트 2,000원 잡아둠”, “토스 카드 10,000원 결제 요청함”
Provider

결제 담당자

provider

결제 수단마다 실제 일을 해 주는 담당자입니다. 포인트 담당자, 토스 담당자, 무통장입금 담당자처럼 나뉩니다.

코드에서는 PaymentProvider 인터페이스를 구현합니다.
Authorize

돈을 잠깐 잡아두기

hold

“이 돈으로 결제할 수 있나?” 확인하는 단계입니다. 포인트는 예약해두고, 카드 결제는 카드사 확인을 받습니다.

아직 최종 확정 전입니다. 실패하면 다시 풀 수 있습니다.
Capture

결제를 확정하기

confirm

잡아둔 돈을 진짜 결제로 확정하는 단계입니다. 이 코드에서는 대부분 자동으로 capture까지 이어집니다.

포인트는 실제 차감되고, 외부 결제는 결제 완료로 기록됩니다.
Kind

provider 종류

kind

여기서 kind는 “친절한”이 아니라 “종류”라는 뜻입니다. 내부 장부인지, 외부 결제망인지 구분합니다.

ledger = 포인트, gateway = 토스/은행/CMS

전체 결제 흐름

가장 흔한 흐름은 “예약표 만들기 → 결제수단 확인 → 돈 잡아두기 → 결제 확정”입니다.

1

쇼핑몰이 예약표를 만듭니다

  • POST /v1/payment-intents
  • 주문 금액을 계산합니다.
  • Intent 상태는 CREATED입니다.
2

사용자가 결제 버튼을 누릅니다

  • POST /:id/confirm
  • 포인트를 쓸지 확인합니다.
  • 카드/은행 결제가 필요한지 봅니다.
3

Wallet이 결제 계획을 세웁니다

  • 포인트 단독인지 봅니다.
  • 외부 결제 단독인지 봅니다.
  • 포인트+외부 복합결제인지 봅니다.
  • Intent 상태는 PROCESSING입니다.
4

담당 provider에게 일을 맡깁니다

  • ProviderRegistry가 담당자를 찾습니다.
  • 포인트면 포인트 원장을 봅니다.
  • 토스면 Toss checkout을 준비합니다.
5

돈을 잡아두거나 대기합니다

  • 성공하면 AUTHORIZED입니다.
  • 사용자 행동이 필요하면 REQUIRES_ACTION입니다.
  • 배치 결과 대기면 PENDING_SETTLEMENT입니다.
6

결제를 확정합니다

  • AutoCaptureService가 자동 확정을 시도합니다.
  • 성공하면 CAPTURED입니다.
  • 이제 주문 쪽에 “결제됨” 이벤트를 보냅니다.

코드가 실제로 하는 일을 더 짧게 보면

위 그림을 실제 서비스 이름과 맞추면 아래처럼 이어집니다.

confirm은 두 단계입니다

Phase 1
DB 안에서 안전하게 준비합니다. Intent를 잠그고, charge 기록을 만들고, 상태를 PROCESSING으로 바꿉니다.
Phase 2
DB 밖에서 provider를 호출합니다. 외부 API는 느리거나 실패할 수 있어서 DB 트랜잭션 밖에서 실행합니다.

ProviderRegistry는 안내 데스크입니다

“TOSS로 결제해줘”라는 요청이 오면 실제 TossPaymentProvider를 찾아줍니다. “POINTS로 결제해줘”라면 PointsPaymentProvider를 찾아줍니다.

method.type → 대문자로 정리 → provider 반환

결제수단별로 뭐가 다른가요?

모두 같은 provider 모양을 쓰지만, 실제 동작은 조금씩 다릅니다.

POINTS

kind: ledger

내부 포인트 장부를 보고 포인트를 먼저 잡아둡니다.

authorize → 포인트 홀드 → capture 때 차감

TOSS

kind: gateway

사용자가 Toss checkout 화면에서 결제를 끝내야 합니다.

authorize → REQUIRES_ACTION → toss-approve

BANK_TRANSFER

kind: gateway

사용자가 계좌로 입금하고, 관리자가 입금 확인을 해줘야 합니다.

authorize → 입금 대기 → 관리자 확인

CMS_BATCH

kind: gateway

은행 출금 신청을 해두고, 나중에 출금 결과를 폴링해서 확인합니다.

출금 신청 → PENDING_SETTLEMENT → poller 성공

포인트 + 외부결제

복합결제

포인트를 먼저 잡아두고, 그 다음 카드/은행/CMS 결제를 진행합니다.

POINTS 먼저 → 외부결제 → 같이 capture

실패하면?

cleanup

외부 결제가 실패하면, 이미 잡아둔 포인트 홀드를 풀려고 시도합니다.

외부 실패 → POINTS cancel → CREATED 복귀

상태값을 쉬운 말로 번역하면

상태는 “지금 결제가 어디까지 왔는지” 알려주는 이름표입니다.

CREATED 예약표만 있음

아직 결제를 시작하지 않았습니다.

PROCESSING 결제 준비 중

Wallet이 결제수단을 보고 provider 호출을 준비합니다.

REQUIRES_ACTION 사람 행동 필요

토스 checkout 완료, 무통장 입금 같은 추가 행동이 필요합니다.

PENDING_SETTLEMENT 외부 결과 기다림

CMS 배치 출금처럼 결과가 나중에 오는 상태입니다.

AUTHORIZED 돈을 잡아둠

결제할 수 있다는 확인을 받았습니다. 이제 확정하면 됩니다.

CAPTURED 결제 확정

실제로 결제가 완료된 상태입니다.

FAILED 실패

결제를 진행하지 못했습니다.

CANCELED 취소

사용자나 시스템이 결제를 취소했습니다.

취소와 환불은 어떻게 달라요?

취소는 “아직 확정 전 일을 멈추기”, 환불은 “이미 확정된 결제를 되돌리기”에 가깝습니다.

취소 Cancel

아직 결제가 끝나기 전이면, 잡아둔 포인트나 외부 결제 승인을 풀어줍니다.

AUTHORIZED → provider.cancel() → CANCELED

환불 Refund

이미 결제가 끝난 뒤에는 refund 기록을 만들고 provider에 환불을 요청합니다.

CAPTURED → refund PENDING → SUCCEEDED

쉬운 설명을 실제 파일 이름에 대응하면 아래와 같습니다.

주요 파일

  • payment-intents.service.ts: create, confirm, capture, cancel 입구
  • confirm.service.ts: 결제 계획 만들고 provider authorize 호출
  • capture.service.ts: authorize된 charge들을 확정
  • provider.registry.ts: provider 찾기와 kind 메타데이터
  • state-transition.rules.ts: 상태가 어디로 이동 가능한지 규칙

주의해서 볼 점

현재 BankTransferAdminService.confirmDeposit()는 intent를 SUCCEEDED로 바꾸려 하지만, 상태 전이 규칙에는 REQUIRES_ACTION → SUCCEEDED가 없습니다.

무통장입금 확인이 실제로 실패한다면, 이 전이 규칙을 AUTHORIZED나 CAPTURED 흐름과 맞추는지 확인해야 합니다.
