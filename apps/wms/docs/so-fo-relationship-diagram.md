# SO-FO 관계 다이어그램 (FulfillmentOrderSource 포함)

## 데이터베이스 구조

```
┌─────────────────┐       ┌─────────────────────────┐       ┌─────────────────┐
│   SalesOrders   │       │ FulfillmentOrderSources │       │FulfillmentOrders│
│                 │       │     (연결 테이블)        │       │                 │
├─────────────────┤       ├─────────────────────────┤       ├─────────────────┤
│ id              │◄──────┤ salesOrderId            │──────►│ id              │
│ channelOrderId  │       │ fulfillmentOrderId      │       │ warehouseId     │

│ shippingAddress │                                         │ labelNo         │

```

## 1. 일반 출고 (1:1)

```
SO-1 ────────────► FOS-1 ────────────► FO-1
(고객 주문)         (연결)            (송장 1개)

┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│ SO-001      │   │ SO: SO-001  │   │ FO-A001     │
│ 채널: 메두사  │───┤ FO: FO-A001 ├───┤ 창고: WH-1  │
│ 상품: 티셔츠  │   │             │   │ 송장: 미발급 │
│ 수량: 2개    │   └─────────────┘   │ 상태: ready │
└─────────────┘                     └─────────────┘
```

## SO 취소 시 처리 흐름

### Case 1: 일반 출고에서 SO 취소
```
SO-1 ──X── FOS-1 ──X── FO-1
(취소)     (삭제)     (취소)
```

### Case 2: 합배송에서 SO 하나만 취소
```
SO-1 ──X── FOS-1 ──X──┐
(취소)     (삭제)      │
                      ├──► FO-1 (계속 진행)
SO-2 ──────FOS-2 ─────┤
                      │
SO-3 ──────FOS-3 ─────┘
```

### Case 3: 송장분리에서 SO 취소
```
SO-1 ──X── FOS-1 ──X── FO-1 (취소)
(취소)     (삭제)
       └── FOS-2 ──X── FO-2 (취소)
           (삭제)
```

## 핵심 장점

1. **SoT 원칙 완전 준수**: SO는 절대 변경되지 않음
2. **유연한 관계 지원**: 모든 패턴 (1:1, 1:N, N:1, N:M) 지원
3. **명확한 취소 처리**: 연결 테이블 삭제만으로 관계 해제
4. **확장 가능성**: 새로운 비즈니스 요구사항 쉽게 수용
5. **데이터 무결성**: Foreign Key로 참조 무결성 보장
