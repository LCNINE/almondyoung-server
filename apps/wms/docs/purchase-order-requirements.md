# 발주 관리 시스템 요구사항

## 비즈니스 배경

### 회사 구조
- **업종**: 미용샵 원장들에게 미용재료 판매하는 소매상
- **창고**: 부천(한국), 중국 - 총 2개 물류창고
- **고객**: 모두 한국에 위치
- **물류 흐름**:
  - 중국 창고: 중국 발주처 → 중국 창고 → 부천 창고 (경유 역할)
  - 부천 창고: 최종 고객 출고 담당

### 발주 프로세스
1. **재고 모니터링**: 안전재고 미만으로 떨어진 상품 식별
2. **발주 결정**: 담당자가 발주 필요 상품을 발주대기리스트에 추가
3. **발주 처리**: 공급업체, 수량, 입고예정일 입력하여 실제 발주
4. **입고 처리**: 입고예정리스트에서 실제 입고 수량 확인 후 처리

## 핵심 요구사항

### 1. 발주대기리스트 (Purchase Cart)
- **목적**: 로그인 사용자별 개인 장바구니 (draft 개념)
- **기능**:
  - 안전재고 미만 상품 중 선택하여 추가
  - 자유로운 추가/제거 가능
  - 편의성을 위한 임시 저장소

### 2. 발주 처리 (Purchase Order)
- **입력 정보**:
  - 공급업체 선택 (재고상품과 발주처는 다대다 관계)
  - 발주 수량
  - 입고 예정일
  - **발주 창고** (실제 발주할 창고)
  - **최종 목적지 창고** (결국 도착할 창고)
- **자동 처리**: 발주 완료시 해당 창고의 입고예정 자동 생성

### 3. 중복 발주 방지
- **추적 대상**:
  - SKU별 발주중 수량
  - SKU별 창고간 이동중 수량
- **표시**: 재고 현황에 "현재재고 + 발주중 + 이동중" 통합 표시
- **방지 로직**:
  - 이미 발주중인 상품 재발주 방지
  - 중국 창고에 충분한 재고가 있어서 창고간 이동으로 해결 가능한 경우 발주 방지
  - 창고간 이동중인 상품 재발주 방지

### 4. 다중 창고 라우팅
- **한국산 제품**: 부천 창고로 직송
- **중국산 제품**: 중국 창고 경유 → 부천 창고
- **최종 목적지**: 항상 부천 창고 (고객 출고 담당)

### 5. 창고간 이동 자동화
- **트리거**: 중국 창고 입고 완료시
- **자동 생성**: 부천 창고로의 이동 작업 자동 생성
- **추적**: 창고간 이동중 수량을 재고 가시성에 반영

### 6. 선적 최적화 (Phase 3)
- **대상**: 중국 → 부천 선적 작업
- **방식**: 박스 단위로 상품 묶음
- **분류 기준**: 로케이션별 (A열, B열, C열...)
- **목적**: 부천 물류팀의 효율적 분류 작업 지원

## 재고 가시성 개선

### 기존 문제점
- 현재 재고만 표시되어 발주중/이동중 수량 미반영
- 중복 발주 위험 존재

### 개선 방향
- **통합 표시**: 현재재고 + 발주중 + 이동중 수량
- **목적지 기준 집계**: 중국에서 발주한 상품도 "부천 입고대기중"으로 분류
- **실시간 업데이트**: 발주/이동 상태 변경시 즉시 반영

## 시스템 설계 원칙

### 확장성 고려
- **하드코딩 금지**: 창고 정보는 모두 파라미터로 처리
- **범용 설계**: 창고 수/구성에 관계없이 작동
- **프론트엔드 기본값**: 필요시 UI에서 기본값 제공

### 데이터 일관성
- **단일 진실의 원천**: 중복 데이터 최소화
- **계산 기반**: 집계 값은 실시간 계산으로 제공
- **트랜잭션 안전성**: 데이터 무결성 보장

## 구현 우선순위

### MVP (즉시 구현)
1. 발주대기리스트 CRUD
2. 기본 발주 처리
3. 중복 발주 방지
4. 재고 가시성 개선

### Phase 2 (단계적 확장)
1. 다중 창고 라우팅
2. 창고간 이동 자동화
3. 이동중 수량 추적

### Phase 3 (고도화)
1. 선적 최적화
2. 박스별 분류 인터페이스
3. 고급 라우팅 규칙

## 기술 고려사항

### API 설계
- RESTful API 원칙 준수
- 창고 ID를 명시적 파라미터로 처리
- 이벤트 기반 자동화 구현

### 데이터베이스
- stockSummary를 view로 전환 검토 (데이터 일관성)
- 발주 상태별 수량 추적 테이블 설계
- 창고간 이동 테이블 추가

### 확장 포인트
- 라우팅 규칙 엔진 (현재는 하드코딩)
- 다국가/다창고 확장 대비 Hub Network 모델
- 비즈니스 규칙 외부화
