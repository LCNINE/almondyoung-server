# 멤버십 구독 시스템 요구사항

## 소개

멤버십 구독 MSA 서버는 사용자의 구독 생명주기를 관리하는 시스템입니다. 결제 처리는 별도 서비스에서 담당하며, 본 시스템은 구독 상태 관리, 플랜 변경, 일시정지/재개, 권한 관리 등의 핵심 비즈니스 로직을 담당합니다. 모든 데이터는 PostgreSQL 데이터베이스에 저장되며, Drizzle ORM을 통해 관리됩니다.

## 요구사항

### 요구사항 1: 구독 상태 조회

**사용자 스토리:** 사용자로서, 내 현재 구독 상태와 이력을 확인할 수 있어야 한다.

#### 승인 기준

1. WHEN 사용자가 현재 구독 상태를 요청하면 THEN 시스템은 활성 구독 정보(티어, 플랜, 다음 결제일, 일시정지 상태)를 반환해야 한다
2. WHEN 사용자가 구독 이력을 요청하면 THEN 시스템은 모든 과거 구독 기록을 시간순으로 반환해야 한다
3. WHEN 활성 구독이 없는 사용자가 조회하면 THEN 시스템은 null 또는 빈 상태를 반환해야 한다
4. WHEN 구독이 일시정지 상태이면 THEN 시스템은 isPaused=true와 pausedAt 정보를 포함해야 한다

### 요구사항 2: 구독 플랜 변경

**사용자 스토리:** 사용자로서, 내 구독 플랜을 업그레이드하거나 다운그레이드할 수 있어야 한다.

#### 승인 기준

1. WHEN 사용자가 더 높은 티어로 업그레이드를 요청하면 THEN 시스템은 즉시 적용하고 정산 금액을 계산해야 한다
2. WHEN 사용자가 더 낮은 티어로 다운그레이드를 요청하면 THEN 시스템은 다음 결제 주기에 적용되도록 예약해야 한다
3. WHEN 일시정지 중인 구독에 플랜 변경을 요청하면 THEN 시스템은 요청을 거부해야 한다
4. WHEN 플랜 변경이 완료되면 THEN 시스템은 관련 이벤트를 발행해야 한다
5. WHEN 업그레이드 시 THEN 시스템은 남은 기간에 대한 차액을 계산해야 한다

### 요구사항 3: 구독 일시정지 및 재개

**사용자 스토리:** 사용자로서, 구독을 일시적으로 정지하고 나중에 재개할 수 있어야 한다.

#### 승인 기준

1. WHEN 사용자가 일시정지를 요청하면 THEN 시스템은 연간 일시정지 한도를 확인해야 한다
2. WHEN 일시정지 한도 내에서 요청하면 THEN 시스템은 구독을 일시정지하고 권한을 중단해야 한다
3. WHEN 일시정지 중인 구독을 재개하면 THEN 시스템은 권한을 복원하고 종료일을 연장해야 한다
4. WHEN 일시정지 기간이 끝나면 THEN 시스템은 자동으로 구독을 재개해야 한다
5. WHEN 일시정지/재개 시 THEN 시스템은 사용량 추적 정보를 업데이트해야 한다

### 요구사항 4: 구독 취소

**사용자 스토리:** 사용자로서, 구독을 취소할 수 있어야 한다.

#### 승인 기준

1. WHEN 사용자가 구독 취소를 요청하면 THEN 시스템은 구독 상태를 CANCELLED로 변경해야 한다
2. WHEN 구독이 취소되면 THEN 시스템은 모든 활성 권한을 종료해야 한다
3. WHEN 구독 취소 시 THEN 시스템은 취소 이벤트를 발행해야 한다
4. WHEN 취소된 구독은 THEN 기존 종료일까지 서비스 이용이 가능해야 한다

### 요구사항 5: 플랜 및 티어 관리

**사용자 스토리:** 시스템 관리자로서, 구독 플랜과 티어를 관리할 수 있어야 한다.

#### 승인 기준

1. WHEN 플랜 목록을 요청하면 THEN 시스템은 모든 활성 플랜 정보를 반환해야 한다
2. WHEN 특정 플랜 상세를 요청하면 THEN 시스템은 플랜과 연관된 티어 정보를 포함해야 한다
3. WHEN 티어 목록을 요청하면 THEN 시스템은 우선순위 순으로 정렬된 티어를 반환해야 한다
4. WHEN 티어별 혜택을 요청하면 THEN 시스템은 해당 티어의 모든 혜택 정보를 반환해야 한다

### 요구사항 6: 관리자 기능

**사용자 스토리:** 시스템 관리자로서, 사용자 구독을 관리하고 감사할 수 있어야 한다.

#### 승인 기준

1. WHEN 관리자가 사용자 구독을 강제 변경하면 THEN 시스템은 변경 사유와 함께 기록해야 한다
2. WHEN 관리자가 크레딧을 지급하면 THEN 시스템은 지급 내역을 추적해야 한다
3. WHEN 관리자가 일시정지 쿼터를 리셋하면 THEN 시스템은 사용량 추적기를 업데이트해야 한다
4. WHEN 감사 로그를 요청하면 THEN 시스템은 모든 중요 작업의 로그를 반환해야 한다
5. WHEN 미발행 이벤트를 조회하면 THEN 시스템은 발행 실패한 이벤트 목록을 반환해야 한다

### 요구사항 7: 이벤트 처리

**사용자 스토리:** 시스템으로서, 모든 구독 변경 사항을 이벤트로 발행하여 다른 서비스와 연동할 수 있어야 한다.

#### 승인 기준

1. WHEN 구독 관련 변경이 발생하면 THEN 시스템은 해당 이벤트를 Kafka로 발행해야 한다
2. WHEN 이벤트 발행이 실패하면 THEN 시스템은 재시도 로직을 실행해야 한다
3. WHEN 이벤트 재발행을 요청하면 THEN 시스템은 특정 이벤트를 다시 발행해야 한다
4. WHEN 이벤트 스트리밍을 요청하면 THEN 시스템은 SSE를 통해 실시간 이벤트를 제공해야 한다

### 요구사항 8: 권한 검증

**사용자 스토리:** 다른 서비스로서, 사용자의 현재 권한을 빠르게 확인할 수 있어야 한다.

#### 승인 기준

1. WHEN 사용자 권한 검증을 요청하면 THEN 시스템은 현재 활성 권한 정보를 반환해야 한다
2. WHEN 벌크 구독 확인을 요청하면 THEN 시스템은 여러 사용자의 구독 상태를 한 번에 반환해야 한다
3. WHEN 권한이 일시정지 상태이면 THEN 시스템은 해당 정보를 명확히 표시해야 한다
4. WHEN 권한이 만료되었으면 THEN 시스템은 만료 상태를 반환해야 한다

### 요구사항 9: 정책 관리

**사용자 스토리:** 시스템 관리자로서, 구독 관련 정책(일시정지 한도 등)을 설정하고 관리할 수 있어야 한다.

#### 승인 기준

1. WHEN 일시정지 요청 시 THEN 시스템은 연간 최대 일시정지 횟수 정책을 확인해야 한다
2. WHEN 정책 위반 시 THEN 시스템은 명확한 오류 메시지와 함께 요청을 거부해야 한다
3. WHEN 정책이 변경되면 THEN 시스템은 새로운 정책을 즉시 적용해야 한다
4. WHEN 티어별 정책이 있으면 THEN 시스템은 사용자의 티어에 맞는 정책을 적용해야 한다
