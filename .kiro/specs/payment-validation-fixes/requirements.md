# Requirements Document

## Introduction

이 문서는 Wallet 결제 시스템의 MVP 출시 전 필수 검증 로직 추가를 위한 요구사항을 정의합니다. QA 리포트에서 식별된 High Priority 이슈 3가지를 해결하여 운영 안정성을 확보합니다.

**배경**:

- 연매출 20억원 규모 이커머스 독립몰 전환
- 현재 스키마 설계는 완벽하나, 비즈니스 로직 검증이 부족
- 세무 가산세 리스크 및 중복 환불 방지 필요

**범위**:

- 부분환불 누적 검증 추가
- 세금계산서 발행 기한 검증 추가
- 환불 API 멱등성 키 적용

**제외 사항**:

- 스키마 변경 없음 (기존 구조 그대로 사용)
- 새로운 API 추가 없음 (기존 API 로직만 강화)
- UI/UX 변경 없음

## Requirements

### Requirement 1: 부분환불 누적 검증

**User Story**: As a 고객센터 직원, I want 환불 요청 시 누적 환불 금액이 결제 금액을 초과하지 않도록 검증되기를, so that 과다 환불로 인한 손실을 방지할 수 있다.

#### Acceptance Criteria

1. WHEN 환불 요청이 들어오면 THEN 시스템은 해당 Intent의 기존 환불 이력을 조회해야 한다
2. WHEN 기존 환불 금액 + 요청 환불 금액이 총 결제 금액을 초과하면 THEN 시스템은 명확한 에러 메시지와 함께 환불을 거부해야 한다
3. WHEN 환불 검증이 실패하면 THEN 에러 메시지는 "이미 환불된 금액", "요청 금액", "총 결제 금액"을 포함해야 한다
4. WHEN 환불 이력 조회 시 THEN 트랜잭션 내에서 `for('update')` 락을 적용하여 동시성 문제를 방지해야 한다

### Requirement 2: 세금계산서 발행 기한 검증

**User Story**: As a 세무 담당자, I want 세금계산서 발행 시 국세청 기한(익월 10일)을 자동으로 검증받기를, so that 가산세를 방지할 수 있다.

#### Acceptance Criteria

1. WHEN 세금계산서 생성 요청이 들어오면 THEN 시스템은 공급일자로부터 익월 10일까지의 기한을 계산해야 한다
2. WHEN 현재 날짜가 발행 기한을 초과했으면 THEN 시스템은 "발행 기한 초과" 에러를 반환해야 한다
3. WHEN 발행 기한 검증이 실패하면 THEN 에러 메시지는 "공급일", "발행 기한"을 포함해야 한다
4. WHEN 수정세금계산서 생성 요청이 들어오면 THEN 시스템은 원본 발행일로부터 6개월 이내인지 검증해야 한다

### Requirement 3: 환불 API 멱등성 키 적용

**User Story**: As a 프론트엔드 개발자, I want 환불 API에 멱등성 키를 적용하여, so that 네트워크 재시도로 인한 중복 환불을 방지할 수 있다.

#### Acceptance Criteria

1. WHEN 환불 API 호출 시 `Idempotency-Key` 헤더가 제공되면 THEN 시스템은 해당 키로 중복 요청을 확인해야 한다
2. WHEN 동일한 멱등성 키로 재요청이 들어오면 THEN 시스템은 기존 환불 결과를 반환해야 한다
3. WHEN 멱등성 키가 없어도 THEN 시스템은 정상적으로 환불을 처리해야 한다 (선택적 기능)
4. WHEN 환불이 완료되면 THEN 시스템은 멱등성 키를 `SUCCESS` 상태로 저장해야 한다
