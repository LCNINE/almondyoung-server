# 요구사항 문서

## 개요

이 기능은 기존 결제 시스템을 확장하여 다양한 PG사 어댑터 구현, 포인트 혼합결제, 그리고 BNPL의 복잡한 정기결제 플로우를 지원합니다. 기존에 구현된 정기결제 로직을 기반으로 하여 각 PG사별 어댑터와 포트를 구현하고, 포인트 혼합결제 기능을 추가합니다. 특히 BNPL은 상품 구매 시 즉시 PG사와 통신하지 않고, 정기결제일에 은행에 총액을 전달하여 2-3일 후 결과를 받는 특수한 플로우를 완성합니다.

## 요구사항

### 요구사항 1: 다중 PG사 어댑터 구현

**사용자 스토리:** 결제 시스템으로서, 표준화된 어댑터를 통해 여러 PG사를 지원하여 고객에게 다양한 결제 옵션을 제공하고 싶습니다.

#### 수락 기준

1. WHEN 결제가 처리될 때 THEN 시스템은 결제 수단 유형에 따라 적절한 PG 어댑터를 선택해야 함
2. WHEN 새로운 PG가 추가될 때 THEN 시스템은 핵심 로직 변경 없이 어댑터 패턴을 통해 지원해야 함
3. WHEN PG 어댑터가 호출될 때 THEN 시스템은 PG사별 인증 및 API 형식을 처리해야 함
4. WHEN PG 응답을 받을 때 THEN 시스템은 이를 표준 내부 형식으로 정규화해야 함
5. IF PG를 사용할 수 없을 때 THEN 시스템은 표준화된 오류 코드로 우아하게 실패해야 함
6. WHEN 여러 PG를 사용할 수 있을 때 THEN 시스템은 결제 수단이나 비즈니스 규칙에 따른 라우팅을 지원해야 함

### 요구사항 2: 포인트 혼합결제 지원

**사용자 스토리:** 고객으로서, 포인트와 다른 결제 수단을 조합하여 결제하여 적립된 포인트를 활용하고 싶습니다.

#### 수락 기준

1. WHEN 혼합결제가 요청될 때 THEN 시스템은 처리 전에 포인트 잔액을 검증해야 함
2. WHEN 혼합결제가 처리될 때 THEN 시스템은 포인트를 먼저 차감한 후 나머지 금액을 결제 수단으로 청구해야 함
3. WHEN 포인트 차감이 실패할 때 THEN 시스템은 결제 수단 청구를 진행하지 않아야 함
4. WHEN 혼합결제가 성공할 때 THEN 시스템은 포인트 사용과 결제 수단 청구를 모두 기록해야 함
5. WHEN 혼합결제가 환불될 때 THEN 시스템은 포인트를 복원하고 결제 수단을 비례적으로 환불해야 함
6. IF 포인트가 부족할 때 THEN 시스템은 명확한 오류 메시지와 함께 거래를 거부해야 함
7. WHEN 포인트 잔액을 확인할 때 THEN 시스템은 실시간 정확성을 보장해야 함

### 요구사항 3: BNPL 정기결제 회원 등록 및 출금동의

**사용자 스토리:** 고객으로서, 적절한 동의를 통해 BNPL 서비스에 등록하여 후불결제 기능을 사용하고 싶습니다.

#### 수락 기준

1. WHEN 고객이 BNPL에 등록할 때 THEN 시스템은 필수 개인정보와 은행계좌 정보를 수집해야 함
2. WHEN BNPL 등록이 제출될 때 THEN 시스템은 자동출금에 대한 명시적 동의를 요구해야 함
3. WHEN 동의가 제공될 때 THEN 시스템은 타임스탬프와 IP 주소와 함께 동의 기록을 저장해야 함
4. WHEN BNPL 계정이 생성될 때 THEN 시스템은 고객 프로필에 따라 초기 신용한도를 설정해야 함
5. WHEN 동의가 철회될 때 THEN 시스템은 자동출금을 비활성화하고 수동결제를 요구해야 함
6. IF 필수 정보가 누락될 때 THEN 시스템은 특정 필드 오류와 함께 등록을 거부해야 함
7. WHEN 동의서가 생성될 때 THEN 시스템은 모든 법적 조건과 출금 일정을 포함해야 함

### 요구사항 4: BNPL 지연 결제 플로우 구현

**사용자 스토리:** BNPL 시스템으로서, 은행 출금이 완료되는 데 2-3일이 걸리는 지연 결제 처리를 관리하여 복잡한 정산 타이밍을 처리하고 싶습니다.

#### 수락 기준

1. WHEN BNPL 구매가 이루어질 때 THEN 시스템은 즉시 은행 통신 없이 승인 기록만 생성해야 함
2. WHEN 월별 정산일이 도래할 때 THEN 시스템은 은행에 출금 요청을 제출해야 함
3. WHEN 은행 출금이 제출될 때 THEN 시스템은 상태를 WITHDRAWAL_REQUESTED로 설정해야 함
4. WHEN 은행이 2-3일 후 응답할 때 THEN 시스템은 결과에 따라 상태를 CAPTURED 또는 FAILED로 업데이트해야 함
5. WHEN 출금이 실패할 때 THEN 시스템은 재시도 로직 또는 수동 개입 프로세스를 트리거해야 함
6. WHEN 출금이 성공할 때 THEN 시스템은 캡처를 완료하고 고객 잔액을 업데이트해야 함
7. IF 고객의 자금이 부족할 때 THEN 시스템은 적절한 고객 알림과 함께 실패한 출금을 처리해야 함

### 요구사항 5: 포인트 서비스 통합

**사용자 스토리:** 결제 시스템으로서, 잔액 확인 및 거래를 위해 포인트 서비스와 통합하여 혼합결제를 정확하게 처리하고 싶습니다.

#### 수락 기준

1. WHEN 포인트 잔액을 조회할 때 THEN 시스템은 고객 ID로 포인트 서비스 API를 호출해야 함
2. WHEN 포인트가 차감될 때 THEN 시스템은 일관성을 보장하기 위해 원자적 트랜잭션을 사용해야 함
3. WHEN 포인트 차감이 실패할 때 THEN 시스템은 부분적인 결제 처리를 롤백해야 함
4. WHEN 포인트가 복원될 때(환불) THEN 시스템은 포인트 서비스 복원 API를 호출해야 함
5. WHEN 포인트 서비스를 사용할 수 없을 때 THEN 시스템은 혼합결제를 우아하게 실패시켜야 함
6. IF 포인트 트랜잭션 충돌이 발생할 때 THEN 시스템은 적절한 오류 코드로 처리해야 함

### 요구사항 6: 결제 방법별 라우팅 로직

**사용자 스토리:** 결제 시스템으로서, 결제 방법과 비즈니스 규칙에 따라 적절한 PG로 결제를 라우팅하여 성공률과 비용을 최적화하고 싶습니다.

#### 수락 기준

1. WHEN 결제 방법이 카드일 때 THEN 시스템은 지정된 카드 PG 어댑터로 라우팅해야 함
2. WHEN 결제 방법이 계좌이체일 때 THEN 시스템은 계좌이체 PG 어댑터로 라우팅해야 함
3. WHEN 결제 방법이 BNPL일 때 THEN 시스템은 즉시 PG 호출 없이 BNPL 전용 처리를 사용해야 함
4. WHEN 혼합결제가 사용될 때 THEN 시스템은 포인트 차감을 먼저 처리한 후 나머지 금액을 적절히 라우팅해야 함
5. WHEN PG 라우팅이 실패할 때 THEN 시스템은 구성된 경우 대체 PG를 시도해야 함
6. IF 적합한 PG가 없을 때 THEN 시스템은 적절한 오류와 함께 결제를 거부해야 함

### 요구사항 7: 정기결제 스케줄링 및 배치 처리

**사용자 스토리:** BNPL 시스템으로서, 월별 정산을 배치로 처리하여 대량의 지연 결제를 효율적으로 처리하고 싶습니다.

#### 수락 기준

1. WHEN 월별 정산일이 도래할 때 THEN 시스템은 모든 대기 중인 BNPL 결제에 대한 배치 작업을 생성해야 함
2. WHEN 배치 처리가 시작될 때 THEN 시스템은 은행별로 결제를 그룹화하고 최적 순서로 처리해야 함
3. WHEN 배치 항목이 처리될 때 THEN 시스템은 출금 요청을 제출하고 상태를 추적해야 함
4. WHEN 배치가 완료될 때 THEN 시스템은 성공/실패 수를 포함한 정산 보고서를 생성해야 함
5. WHEN 배치 처리가 실패할 때 THEN 시스템은 실패한 항목에 대한 재시도 메커니즘을 지원해야 함
6. IF 배치 처리가 중단될 때 THEN 시스템은 마지막 성공한 체크포인트에서 재개해야 함

### 요구사항 8: 어댑터별 에러 처리 및 재시도

**사용자 스토리:** 결제 시스템으로서, 각 PG 어댑터에 대한 강력한 오류 처리를 통해 일시적 실패가 거래 손실로 이어지지 않도록 하고 싶습니다.

#### 수락 기준

1. WHEN PG 타임아웃이 발생할 때 THEN 시스템은 지수 백오프 재시도 로직을 구현해야 함
2. WHEN PG가 일시적 오류를 반환할 때 THEN 시스템은 구성된 최대 시도 횟수까지 재시도해야 함
3. WHEN PG가 영구적 오류를 반환할 때 THEN 시스템은 재시도 없이 즉시 실패해야 함
4. WHEN 재시도 한계를 초과할 때 THEN 시스템은 거래를 실패로 표시하고 관리자에게 알려야 함
5. WHEN 네트워크 오류가 발생할 때 THEN 시스템은 타임아웃과 연결 실패를 구분해야 함
6. IF PG 어댑터가 예외를 던질 때 THEN 시스템은 오류 세부사항을 로그하고 표준화된 오류 코드를 반환해야 함

### 요구사항 9: 혼합결제 트랜잭션 무결성

**사용자 스토리:** 결제 시스템으로서, 혼합결제의 트랜잭션 무결성을 보장하여 부분적 실패가 시스템을 일관성 없는 상태로 만들지 않도록 하고 싶습니다.

#### 수락 기준

1. WHEN 혼합결제가 시작될 때 THEN 시스템은 분산 트랜잭션 패턴을 사용해야 함
2. WHEN 포인트 차감은 성공하지만 결제가 실패할 때 THEN 시스템은 자동으로 포인트를 복원해야 함
3. WHEN 결제는 성공하지만 포인트 차감이 실패할 때 THEN 시스템은 결제를 무효화해야 함
4. WHEN 두 작업이 모두 성공할 때 THEN 시스템은 두 트랜잭션을 원자적으로 커밋해야 함
5. WHEN 롤백이 필요할 때 THEN 시스템은 포인트와 결제 작업이 모두 되돌려지도록 보장해야 함
6. IF 혼합결제 중 시스템이 크래시될 때 THEN 시스템은 재시작 시 복구하고 완료하거나 롤백해야 함

### 요구사항 10: 실시간 결제 상태 추적

**사용자 스토리:** 고객 서비스 담당자로서, 지연된 BNPL 처리를 포함한 실시간 결제 상태를 추적하여 고객에게 정확한 정보를 제공하고 싶습니다.

#### 수락 기준

1. WHEN 결제 상태를 조회할 때 THEN 시스템은 BNPL 전용 상태를 포함한 현재 단계를 반환해야 함
2. WHEN BNPL 출금이 대기 중일 때 THEN 시스템은 예상 완료 날짜를 표시해야 함
3. WHEN 혼합결제가 처리될 때 THEN 시스템은 포인트 사용과 결제 수단 청구의 분석을 표시해야 함
4. WHEN 결제 이벤트가 발생할 때 THEN 시스템은 실시간으로 상태를 업데이트해야 함
5. WHEN 관리자가 결제를 조회할 때 THEN 시스템은 PG 상호작용을 포함한 완전한 타임라인을 표시해야 함
6. IF 상태 조회가 실패할 때 THEN 시스템은 타임스탬프와 함께 마지막으로 알려진 상태를 반환해야 함
