# 멤버십 시스템 아키텍처 문서

## 개요

이 문서는 NestJS 기반의 마이크로서비스 아키텍처로 구축된 멤버십 시스템의 전체적인 구조, API 설계, 데이터 흐름을 설명합니다.

## 시스템 아키텍처

### 전체 구조

```
멤버십 시스템
├── Controller Layer (API 엔드포인트)
├── Service Layer (비즈니스 로직)
├── Data Access Layer (Drizzle ORM)
└── Database (PostgreSQL)
```

#### 2. 서비스 레이어 (Service Layer)

##### 2.1 SubscriptionService

**역할**: 구독 생명주기 관리

**주요 메서드**:

- `getCurrentSubscriptionDetails(userId)`: 현재 구독 상태 조회
- `createSubscription(userId, planId)`: 새 구독 생성
- `upgradeSubscription(userId, newPlanId)`: 구독 업그레이드
- `cancelSubscription(userId, reason)`: 구독 취소
- `getSubscriptionHistory(userId)`: 구독 이력 조회

**비즈니스 로직**:

- 트랜잭션을 통한 데이터 일관성 보장
- 이벤트 배치 생성으로 변경 이력 추적
- 계약(Contract)과 권한(Entitlement) 동시 관리

##### 2.2 EntitlementService

**역할**: 사용자 구독 권한 관리

**주요 메서드**:

- `createEntitlement()`: 새 권한 생성
- `getUserEntitlement(userId)`: 사용자 권한 조회
- `adjustEntitlement()`: 권한 기간 조정 (연장/차감)
- `extendEntitlement()`: 권한 연장

**특징**:

- 권한 생성 시 기존 활성 권한 자동 종료
- 일시정지 상태 관리
- 관리자 액션에 대한 이력 추적

##### 2.3 PlanService

**역할**: 플랜 및 티어 관리

**주요 메서드**:

- `getAllPlans()`: 모든 활성 플랜 조회
- `getPlanDetails(planId)`: 플랜 상세 정보
- `getAllTiers()`: 모든 티어 조회
- `getPlansByTier(tierId)`: 티어별 플랜 조회
- `getTierBenefits(tierId)`: 티어 혜택 조회
- 관리자용: `createTier()`, `updateTier()`, `createPlan()`, `updatePlan()`, `deactivatePlan()`

##### 2.4 PauseService

**역할**: 구독 일시정지 관리

**주요 메서드**:

- `pauseSubscription()`: 구독 일시정지
- `resumeSubscription()`: 구독 재개
- `getPauseHistory()`: 일시정지 이력

**특징**:

- 일시정지 기간만큼 구독 종료일 자동 연장
- `pauseEntitlementVoids` 테이블로 조정 이력 추적

##### 2.5 AdminOperationsService

**역할**: 관리자 작업 오케스트레이션

**특징**:

- 각 도메인 서비스에 작업 위임
- 비즈니스 로직 최소화
- 관리자 권한 검증

##### 2.6 PolicyManagementService

**역할**: 구독 정책 CRUD 관리

**주요 메서드**:

- `createPolicy()`: 정책 생성
- `updatePolicy()`: 정책 수정
- `getAllPolicies()`: 정책 목록 조회
- `deactivatePolicy()`: 정책 비활성화

## 데이터베이스 스키마

### 핵심 테이블 구조

#### 1. 기본 엔티티

```sql
-- 사용자
users (id, created_at, updated_at)

-- 티어 (구독 등급)
tiers (id, code, priority_level, created_at, updated_at)

-- 플랜 (구독 상품)
plan (id, tier_id, price, duration_days, currency, trial_days, is_active, created_at, updated_at)
```

#### 2. 구독 관리

```sql
-- 구독 계약
subscription_contracts (id, user_id, plan_id, next_billing_date, lead_days, is_voided, voided_at, reason, created_at)

-- 구독 권한
subscription_entitlement (id, user_id, tier_id, starts_at, ends_at, created_at, closed_at, is_current, source_batch_id, closed_batch_id, paused_at)
```

#### 3. 이벤트 및 이력

```sql
-- 이벤트 배치
event_batches (id, type, admin_id, effective_date, created_at)

-- 일시정지 기간
pause_periods (id, user_id, starts_at, ends_at, reason, created_at)

-- 일시정지로 인한 권한 조정
pause_entitlement_voids (id, pause_id, entitlement_id, original_ends_at, adjusted_ends_at, created_at)
```

#### 4. 정책 관리

```sql
-- 구독 정책
subscription_policies (id, rule_type, rule_value, tier_id, is_active, valid_from, valid_until, created_at, updated_at)
```

### 데이터 흐름

#### 1. 구독 생성 플로우

```
1. 사용자 구독 요청
2. 기존 활성 구독 확인
3. 플랜 유효성 검증
4. 트랜잭션 시작
   - 이벤트 배치 생성
   - 구독 계약 생성
   - 구독 권한 생성
5. 트랜잭션 커밋
```

#### 2. 구독 업그레이드 플로우

```
1. 현재 구독 상태 확인
2. 새 플랜 유효성 및 등급 검증
3. 트랜잭션 시작
   - 이벤트 배치 생성
   - 기존 계약 플랜 업데이트
   - 기존 권한 종료
   - 새 권한 생성
4. 트랜잭션 커밋
```

#### 3. 일시정지 플로우

```
1. 정책 검증 (PolicyGuard)
2. 현재 활성 권한 확인
3. 트랜잭션 시작
   - 이벤트 배치 생성
   - 일시정지 기간 레코드 생성
   - 권한 조정 내역 기록
   - 기존 권한 종료
   - 연장된 종료일로 새 권한 생성
4. 트랜잭션 커밋
```

## 보안 및 인증

### 현재 상태

- **개발 환경**: `DevAuthGuard` 사용 (임시)
- **프로덕션 대응 필요**: JWT 기반 인증으로 교체 예정

### 권한 관리

- **PolicyGuard**: 구독 정책 검증
- **관리자 권한**: 모든 관리자 API에 인증 가드 적용

## 주요 특징

### 1. 이벤트 기반 아키텍처

- 모든 중요한 변경사항을 `event_batches`로 추적
- 감사 로그와 연동하여 완전한 이력 관리

### 2. 트랜잭션 일관성

- 모든 복합 작업을 데이터베이스 트랜잭션으로 처리
- 데이터 무결성 보장

### 3. 유연한 정책 시스템

- JSON 기반 정책 규칙 저장
- 티어별 또는 전체 적용 가능한 정책

### 4. 권한 이력 관리

- 권한 변경 시 기존 권한 종료 후 새 권한 생성
- 완전한 변경 이력 보존

## 에러 처리

### 커스텀 예외 클래스

- `SubscriptionNotFoundException`: 구독을 찾을 수 없음
- `ActiveSubscriptionExistsException`: 이미 활성 구독 존재
- `PlanNotFoundException`: 플랜을 찾을 수 없음
- `EntitlementNotFoundException`: 권한을 찾을 수 없음
- `BadRequestException`: 잘못된 요청

### 예외 필터

- `SubscriptionExceptionFilter`: 구독 관련 예외 통합 처리

## 개발 고려사항

### 1. 확장성

- 마이크로서비스 아키텍처로 독립적 배포 가능
- 도메인별 서비스 분리로 유지보수성 향상

### 2. 데이터 일관성

- Drizzle ORM 트랜잭션 활용
- 이벤트 소싱 패턴으로 변경 이력 추적

### 3. 성능 최적화

- 인덱스 최적화 필요 (user_id, is_current 등)
- 캐싱 전략 고려 (플랜 정보, 티어 정보)

### 4. 모니터링

- 감사 로그 시스템으로 모든 변경사항 추적
- 관리자 액션 별도 로깅

## 향후 개선 사항

1. **인증 시스템**: DevAuthGuard → JWT 인증으로 전환
2. **알림 시스템**: 구독 상태 변경 시 사용자 알림
3. **결제 연동**: 실제 결제 시스템과 연동
4. **성능 모니터링**: APM 도구 연동
5. **캐싱 레이어**: Redis를 통한 조회 성능 개선
